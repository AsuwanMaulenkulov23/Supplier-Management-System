# Задача 1 — Алгоритм подбора поставщика

Блок-схема: [`flowchart.svg`](flowchart.svg) (рядом — PNG-превью `flowchart.png`).

Алгоритм выполняется **для каждого заказа** из Таблицы 2 — как при первичной
загрузке seed-данных, так и при создании нового заказа в веб-приложении.
В коде он вынесен в чистый, не зависящий от БД модуль
[`algorithm.py`](../algorithm.py) (функция `select_supplier`), что позволяет
покрыть его тестами и переиспользовать в обоих сценариях.

## Описание по шагам

1. **Вход.** На вход поступает заказ: фрукт, количество (кг), дата заказа и срок
   поставки клиенту.

2. **Доступное время.** Считаем, сколько у нас дней, чтобы товар оказался у нас
   *до* поставки клиенту:
   `доступное время = срок поставки клиенту − дата заказа`.
   Пример из ТЗ: заказ 01.05, поставка клиенту 03.05 → **2 дня**.

3. **Поиск по фрукту.** Находим всех поставщиков нужного фрукта (Таблица 1).
   Если таких нет — заказ сразу уходит в **«Требует ручного разбора»**.

4. **Фильтр по остатку.** Оставляем только тех поставщиков, у кого остаток на
   складе ≥ количества в заказе (заказ покрывается одним поставщиком —
   разбиение между несколькими вынесено в необязательные задачи).
   Если не осталось никого — **«Требует ручного разбора»**.

5. **Фильтр по сроку.** Оставляем тех, кто успевает: срок поставки поставщика
   (дней) ≤ доступного времени. Поставщик со сроком «3 дня» при доступных 2 днях
   уже **не подходит**. Если не осталось никого — **«Требует ручного разбора»**.

6. **Выбор лучшего.** Из оставшихся берём поставщика с **минимальной ценой
   закупки**. При равной цене для детерминированности выбираем того, кто быстрее,
   затем — с меньшим id.

7. **Расчёт финансов.** Применяем минимальную наценку 25%:
   - `цена продажи = цена закупки × 1.25`;
   - `сумма заказа = цена продажи × количество`.

8. **Списание склада.** После «закрытия» заказа уменьшаем остаток выбранного
   поставщика на использованное количество. Поэтому последующие заказы по тому
   же фрукту видят уже актуальные остатки (заказы обрабатываются
   последовательно).

9. **Статус.** Заказу проставляется **«Подобран поставщик»**. Если на любом из
   шагов 3–5 кандидатов не осталось — **«Требует ручного разбора»** с указанием
   причины (нет фрукта / не хватает остатка / никто не успевает).

## Бизнес-правила из ТЗ и где они реализованы

| Правило | Реализация |
| --- | --- |
| Минимальная наценка 25% | `MIN_MARKUP = 0.25`, шаг 7 |
| Логистикой пренебрегаем | в расчёте суммы нет статей логистики |
| Срок поставки критичен | шаги 2 и 5 |
| Списание склада после закрытия | шаг 8 (и возврат остатка при удалении/редактировании заказа) |
| Нет подходящих → ручной разбор | шаги 3–5, статус «Требует ручного разбора» |
