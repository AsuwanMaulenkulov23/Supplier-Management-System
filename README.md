# ФрутСнаб — подбор поставщиков и финансы

Веб-приложение по тестовому заданию: автоматический подбор поставщика для каждого
заказа, расчёт финансов и ведение обеих таблиц сотрудниками с разными ролями.

- **Задача 1** (алгоритм + блок-схема) — см. [`docs/algorithm.md`](docs/algorithm.md)
  и [`docs/flowchart.svg`](docs/flowchart.svg).
- **Задача 2** (веб-приложение) — Flask + SQLite + ванильный фронтенд.
- **Задача 3** (роли и права) — три роли, разграничение на бэкенде и в интерфейсе.

## Стек

- **Бэкенд:** Python 3 + Flask, база — SQLite (стандартная библиотека `sqlite3`,
  без внешних ORM).
- **Фронтенд:** один HTML + ванильный JS + CSS (без сборки). Автообновление
  данных — polling каждые 15 секунд.
- **Seed:** при первом запуске данные подгружаются из приложенных Excel-файлов
  (`data/suppliers.xlsx`, `data/orders.xlsx`) через `openpyxl`.

## Запуск (локально)

Нужен Python 3.10+.

```bash
cd fruit-supply
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt
python app.py
```
 
Откройте **http://127.0.0.1:5000**.

При первом запуске автоматически создаётся `app.db`, заполняется из Excel и
прогоняется алгоритм подбора по всем заказам. Чтобы пересоздать данные с нуля:

```bash
python seed.py --reset
```

## Тестовые аккаунты

| Роль | Логин | Пароль |
| --- | --- | --- |
| Закупщик | `buyer` | `BUYER` |
| Менеджер продаж | `manager` | `MANAGER` |
| Руководитель | `director` | `DIRECTOR` | 

На экране входа есть кнопки-подсказки, подставляющие эти данные в один клик.

## Роли и права (Задача 3)

| Роль | Поставщики | Заказы | Финансы | История |
| --- | --- | --- | --- | --- |
| **Закупщик** | CRUD | только чтение | нет доступа | нет |
| **Менеджер продаж** | только чтение | CRUD | только финансы по заказам (без общей сводки компании) | нет |
| **Руководитель** | CRUD | CRUD | полная сводка компании | да |

Права проверяются **на сервере** (декоратор `@require` в `app.py`), а не только
прячутся в интерфейсе: запрос «не своей» операции вернёт `403`, даже если послать
его в обход UI. Это проверено в `_smoke.py`-сценарии при разработке.

> **Про «свои финансы по заказам».** В исходных данных у заказов нет привязки к
> конкретному менеджеру, поэтому менеджер видит финансовую разбивку по всем
> заказам, но **без** агрегированной сводки компании (общая выручка/прибыль,
> разрезы по фруктам и клиентам) — ровно та граница, которую задаёт ТЗ. Если бы
> заказы привязывались к автору, фильтрация «только свои» делается добавлением
> `WHERE owner_id = ?`.

## Функционал

Обязательное:

- Авторизация по логину/паролю (сессии, пароли хранятся хешированными).
- Три роли с разными правами (см. таблицу выше).
- Страница **«Поставщики»** — таблица 1 с добавлением/редактированием/удалением.
- Страница **«Заказы»** — таблица 2; при создании заказа алгоритм автоматически
  подбирает поставщика, считает цену продажи и сумму, проставляет статус
  («Подобран поставщик» / «Требует ручного разбора»). Видна причина ручного разбора.
- Страница **«Финансы»** — общая выручка, общая прибыль, прибыль по фруктам и по
  клиентам, плюс детализация по заказам.
- Автообновление у всех пользователей (polling каждые 15 секунд).
- Seed из приложенных Excel-файлов при старте.

Реализованные необязательные пункты (бонусы):

- **История изменений** — кто и когда менял запись (страница «История» у
  руководителя).
- **Возврат остатка** при редактировании/удалении заказа (склад пересчитывается
  корректно).
- **Графики** на странице финансов — горизонтальные бары прибыли по фруктам и
  клиентам.


Покрыты: пример из ТЗ (доступное время), выбор самого дешёвого среди успевающих,
выбор самого дешёвого когда успевают все, ручной разбор по остатку, по сроку и по
